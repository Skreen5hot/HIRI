## ADR-001: Use JSON-LD Deterministic Service Template

**Date:** [TODAY]

**Decision:** Adopt the JSON-LD Deterministic Service Template as the base architecture.

**Context:** We need a service that produces deterministic, reproducible transformations on structured data. The template provides a pure kernel with spec tests, layered boundaries (kernel/composition/adapters), and zero runtime dependencies.

**Consequences:**
- All transformation logic lives in `src/kernel/transform.ts` as pure functions
- Kernel MUST NOT perform I/O, reference time, randomness, or environment state
- Infrastructure (HTTP, persistence, scheduling) lives in `src/adapters/`
- Spec tests (determinism, no-network, snapshot, purity) MUST pass before any merge

## ADR-002: Use @noble/ed25519 for Ed25519 Cryptography

**Date:** 2026-03-03

**Decision:** Adopt `@noble/ed25519` as a runtime dependency for Ed25519 signing and verification.

**Context:** HIRI Protocol v2.1.0 requires Ed25519 signatures for resolution manifests. Web Crypto API does not yet support Ed25519 universally across all target runtimes. Alternatives considered: custom implementation (rejected — cryptography should not be hand-rolled), TweetNaCl.js (older, less maintained), deferring signing to a later milestone (rejected — signing is core to the Verifiable Atom).

**Consequences:**
- First runtime dependency in `package.json` — breaks the "zero runtime deps" property of the template
- Unblocks Milestone 1 steps 3c–3i (keypair generation, signing, verification)
- `@noble/ed25519` is audited, pure JS, no native bindings — preserves edge-canonical execution
- Must verify the library works in both Node.js and browser contexts

## ADR-003: Timestamps as Input Parameters (No Clock in Kernel)

**Date:** 2026-03-03

**Decision:** Timestamps are passed into the kernel as input data, not generated by the kernel.

**Context:** HIRI resolution manifests require `created` and `expires` fields. Two approaches were evaluated: (A) timestamp as a function parameter supplied by the caller, (B) injectable clock abstraction defaulting to `Date.now()`. Option B would violate the CLAUDE.md determinism rule which explicitly forbids `Date.now()` in kernel code, and would require weakening the purity checker.

**Consequences:**
- Kernel remains a pure function: same input always produces same output
- `npm run test:purity` and determinism spec tests require no exceptions or workarounds
- Callers (adapters, CLI) are responsible for supplying timestamps
- Kernel cannot validate temporal constraints (e.g., "not expired") — that logic belongs in adapters
- Tests are trivially deterministic — pass the same timestamp, get the same result

## ADR-004: Minimal StorageAdapter Interface (Two Methods)

**Date:** 2026-03-03

**Decision:** The `StorageAdapter` interface has exactly two methods: `get(hash): Promise<Uint8Array | null>` and `has(hash): Promise<boolean>`. No `put`, no indexing, no queries.

**Context:** v1.2 Milestone 3 defines the abstracted resolver — `Resolve(URI, StorageAdapter, Config) → VerifiedContent`. The storage interface must be minimal enough that any content-addressed backend can implement it (in-memory, filesystem, IPFS, S3). Alternatives considered: a richer interface with `put`, `list`, `getLatest` (rejected — conflates resolution with publishing and indexing, which are separate concerns).

**Consequences:**
- Publishing (`put`) is a method on concrete adapters (e.g., `InMemoryStorageAdapter.put()`), not on the interface
- Indexing (`getLatest`, `listVersions`, `listBranches`) belongs in a separate `ManifestIndex` interface (deferred)
- `StorageAdapter.get()` makes no hash verification guarantee — callers MUST verify returned bytes
- Any storage backend that can retrieve bytes by hash can implement the interface
- The resolver bridge function (`manifestFetcherFromStorage`) handles deserialization + hash verification

## ADR-005: Oxigraph WASM for In-Memory RDF Store and SPARQL Engine

**Date:** 2026-03-03

**Decision:** Adopt `oxigraph` (WASM build) as a runtime dependency for in-memory RDF graph storage and SPARQL 1.1 query execution.

**Context:** Milestone 4 ("The Storeless Oracle") requires loading verified JSON-LD content into a local RDF index and querying it with SPARQL — without a graph database server. Oxigraph provides a full SPARQL 1.1 engine in ~2–3MB of WASM. It does not apply RDFS inference by default, which is ideal for the MVP's `entailmentMode: "none"`. Alternatives considered: `n3.js` (Turtle/N-Triples parser, no built-in SPARQL), custom SPARQL evaluator (rejected — too large a scope for a single milestone). Follows ADR-002 pattern: WASM loads in the adapter layer (`src/adapters/rdf/`), kernel defines `RDFIndex` and `SPARQLEngine` interfaces only.

**Consequences:**
- Second runtime dependency in `package.json`
- Unblocks Milestone 4 tests 4.1–4.11
- Oxigraph WASM does NOT parse JSON-LD — requires a separate JSON-LD → N-Quads conversion step (see ADR-006)
- The single `OxigraphRDFStore` adapter class implements both `RDFIndex` and `SPARQLEngine` — no type-unsafe downcasts
- Must verify WASM loads correctly in Node.js ESM context

## ADR-006: jsonld Library for JSON-LD to N-Quads Conversion

**Date:** 2026-03-03

**Decision:** Adopt `jsonld` as a runtime dependency for converting JSON-LD documents to N-Quads format before loading into Oxigraph.

**Context:** Oxigraph WASM supports N-Triples, N-Quads, Turtle, TriG, and RDF/XML — but NOT JSON-LD. The HIRI protocol's canonical content format is `application/ld+json`. The `jsonld` library is the canonical JSON-LD processor for JavaScript, providing `jsonld.toRDF()` which converts JSON-LD to N-Quads. Alternatives considered: converting all fixtures to Turtle (rejected — breaks the protocol's content format contract), pre-converting at test setup only (rejected — the adapter must handle the protocol's canonical format).

**Consequences:**
- Third runtime dependency in `package.json`
- The `jsonld` library lives in the adapter layer (`src/adapters/rdf/`), not the kernel
- Remote context fetches are explicitly blocked via a custom `documentLoader` that throws on any URL — all fixtures use inline `@context` objects
- Test 4.11 verifies no network activity during RDF processing
- Future fixtures must use inline contexts to avoid remote fetches

## ADR-007: Wrap Existing Functions for Key Lifecycle (M5)

**Date:** 2026-03-03

**Decision:** Create new lifecycle-aware functions (`verifyManifestWithKeyLifecycle`, `verifyChainWithKeyLifecycle`) alongside existing bare-key functions rather than modifying them.

**Context:** Milestone 5 adds key rotation/revocation awareness to signature verification. The existing `verifyManifest()` and `verifyChain()` are called throughout M1–M4 tests (68 tests) with bare public keys. Two approaches were evaluated: (A) modify existing functions to accept an optional `KeyDocument` parameter, (B) create new functions that wrap the existing ones with lifecycle checks. Option A risks breaking 68 existing tests and introduces branching complexity in functions that currently have clean single-responsibility signatures. Option B adds slight indirection but preserves backward compatibility without any test modifications.

**Consequences:**
- All 68 existing M1–M4 tests pass without modification
- `verifyManifest(manifest, publicKey, crypto)` signature unchanged — still works for bare-key verification
- `verifyManifestWithKeyLifecycle(manifest, keyDocument, verificationTime, crypto)` wraps the existing function with temporal key status checks
- `verifyChainWithKeyLifecycle()` walks entire chain without short-circuit (required for partial validity reporting), while `verifyChain()` retains early-exit behavior
- `resolve()` conditionally routes to lifecycle-aware functions when `keyDocument` is provided in `ResolveOptions` — all new fields are optional
- Two code paths in the resolver: lifecycle-aware (M5+) and bare-key (M1–M4), selected at runtime by presence of `keyDocument`

## ADR-011: RDF Patch Design and canonicalizeNQuads Interface Extension

**Date:** 2026-03-06

**Decision:** Extend the `Canonicalizer` interface with an optional `canonicalizeNQuads` method for re-canonicalization of patched N-Quads. RDF Patch operations are pure kernel functions in `src/kernel/rdf-patch.ts`. The `verifyDelta` function dispatches by format: JCS uses JSON Patch, URDNA2015 uses RDF Patch with post-patch re-canonicalization.

**Context:** The spec requires RDF Patch format for URDNA2015-profile deltas. After applying add/remove operations to a quad set, the result must be re-canonicalized via URDNA2015. The `appliesTo` hash check in `verifyDelta` is format-aware: JCS hashes raw bytes, URDNA2015 canonicalizes first then hashes the canonical form.

**Consequences:**
- `Canonicalizer.canonicalizeNQuads?()` is optional (JCSCanonicalizer does not implement it)
- RDF Patch operations (`parseNQuads`, `serializeNQuads`, `applyRDFPatch`) are pure kernel functions
- Remove-of-nonexistent is a no-op (RDF set semantics); atomicity enforced at pipeline level
- `verifyDelta` `appliesTo` check is profile-aware: deferred to post-canonicalization for URDNA2015
- Chain walker threads `canonicalizer` and `documentLoader` to `verifyDelta`
