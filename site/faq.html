HIRI Protocol Demo FAQ Developer Guide

Frequently Asked Questions

HIRI (Hash-IRI) is a decentralized, edge-first, verifiable knowledge system. These questions address common technical and business concerns.

General Overview

What is HIRI?
A decentralized protocol for verifiable, privacy-aware knowledge. Data is JSON-LD, cryptographically signed, content-addressed, and organized into tamper-evident version chains. Content can be public, encrypted for specific recipients, selectively disclosed at the RDF statement level, or held privately — all without centralized infrastructure.
Is HIRI a product, a platform, or a protocol?
A specification with a reference implementation. Two specs: core protocol (v3.1) and Privacy Extension (v1.4.1). The reference implementation has 323 tests, an interactive browser demo, and a developer guide. Deployments can be browser apps, CLI tools, desktop apps, or servers — none required by the spec.
How does HIRI differ from blockchain?
HIRI uses hash-linked chains but requires no consensus, no global state, and no mining. Each authority manages its own chain. Verification is local and deterministic — no network needed.
How does HIRI differ from W3C Verifiable Credentials?
VCs focus on claims about subjects ("this person has a degree"). HIRI verifies arbitrary JSON-LD documents — not just credentials. HIRI adds version chains, five privacy modes, cross-mode chain walking, and statement-level selective disclosure of individual RDF triples.

Architecture

Does HIRI require servers?
No. The demo — including all five privacy modes, AES-256-GCM encryption, X25519 key agreement, and SPARQL querying — runs entirely in your browser with zero network requests.
What is "Edge-Canonical"?
The system must function fully on a single machine. Anything that can't is infrastructure, not core logic, and must be optional.
What is the layer architecture?
Three layers. Layer 0 (Kernel): pure computation — signing, chaining, resolution. No I/O, no dependencies. Layer 1 (Adapters): crypto, storage, RDF engines. Layer 2 (Privacy): five privacy modes composing with Layers 0 and 1 without modifying either. An automated purity checker enforces boundaries. Zero kernel files were changed across the entire privacy implementation.
What storage backends are supported?
Anything implementing get(hash) and has(hash). Reference includes in-memory, filesystem, and latency-simulated adapters. IndexedDB, S3, IPFS — all compatible.

Privacy Modes

What are the five privacy modes?
Mode 1 — Proof of Possession: Proves you hold data without revealing it. Content is never stored.

Mode 2 — Encrypted Distribution: AES-256-GCM encryption with per-recipient X25519 key agreement. Unauthorized parties verify the manifest but can't read content.

Mode 3 — Selective Disclosure: Canonicalize JSON-LD into individual RDF statements. Disclose some, withhold others. Each gets a salted hash and HMAC proof.

Mode 4 — Anonymous Publication: One-time ephemeral keypair, destroyed after signing. Two publications are computationally unlinkable. Composes with any content visibility.

Mode 5 — Attestation: Third party vouches for a property of someone else's data. Dual-signature verification. Trust reported as full, partial, or unverifiable.
Can I encrypt content for specific recipients?
Yes. Mode 2 generates a random content encryption key, encrypts with AES-256-GCM, then wraps the key separately for each recipient via X25519 key agreement. Tab E.2 lets you encrypt for multiple recipients and resolve as each one — including an unauthorized party.
What is selective disclosure?
RDF-level access control. A personnel record can disclose name and title publicly, email to partners, and birthdate to nobody. Each statement is independently verifiable. The disclosed statements form a valid RDF subgraph loadable into SPARQL.
Can I publish anonymously?
Yes. Ephemeral authorities are one-time and unlinkable. Pseudonymous authorities are persistent and linkable by design. Both compose with any content visibility — an anonymous whistleblower can publish selectively disclosed evidence.
What if I lose my decryption key?
The data is permanently opaque to you. No password reset, no admin override. The publisher can re-encrypt for new recipients if they still have the plaintext, but that requires action. Treat recipient key management with the same rigor as any PKI deployment.
How does attestation trust work?
Dual-signature: attestor's signature + subject authority's signature. Full: both verified, key active. Partial: subject unavailable or key revoked but subject verified. Unverifiable: attestor sig fails or key revoked with no subject. Stale attestations (past validUntil) are flagged but not invalid.
Can privacy modes change across versions?
Yes, but only toward less privacy. PoP → Encrypted → Public is valid. Public → anything private is not — you can't unpublish. The chain walker uses getLogicalPlaintextHash() to maintain content identity even when the storage format changes between modes.

Security & Keys

How does key rotation work?
Dual-signature proof: old key and new key both sign the rotation claim. Grace period keeps old-key signatures valid for a configured duration. After expiry, old-key signatures are rejected. Try the Mock Clock slider in Tab A.
What cryptographic primitives are used?
Signing: Ed25519. Hashing: SHA-256. Encryption: AES-256-GCM. Key agreement: X25519 ECDH → HKDF-SHA256 → AES-GCM wrap. HMAC: HMAC-SHA256. Canonicalization: JCS (JSON) or URDNA2015 (RDF). Ed25519 is for signing; X25519 is for key agreement — separate key types, no cross-usage.
What does "verified: true, decryption-failed" mean?
The manifest is authentic and untampered (signature and chain are valid). You just don't have the right key to read the content. Verification and decryption are orthogonal. This is not an error — it's the expected result for unauthorized parties.
Are private keys protected in the implementation?
Yes. Content encryption keys, ephemeral private keys, and HMAC keys are explicitly zeroed after each operation. No privileged access, background daemons, or system-level services required — everything runs in the browser sandbox or user-level Node process.

Knowledge & Semantics

What role does JSON-LD play?
Canonical representation for all knowledge artifacts. Provides semantic interoperability (terms resolve to URIs), graph structure (loadable into RDF stores), and deterministic canonicalization (URDNA2015 for selective disclosure, JCS for general signing).
Why does entailment default to "none"?
Deterministic, auditable results. If SoftwareArchitect subClassOf Person, querying for Person returns zero results under "none" — inference must be explicitly requested. Tab C demonstrates this boundary.
Can disclosed statements be queried with SPARQL?
Yes. Mandatory N-Quads from selective disclosure form a valid RDF subgraph. Load into Oxigraph, query with SPARQL. Withheld statements are simply absent — no false positives, no schema violations.

Advanced Topics

How do deltas work?
Version updates include a JSON Patch (or RDF Patch for URDNA2015 profiles) alongside full content. The resolver tries the delta first — faster and smaller. If the patched hash doesn't match, it falls back to full content. Deltas are an optimization, not a trust boundary.
What about the dictionary attack on withheld statements?
For low-entropy fields (8 blood types, ~36K birth dates), an attacker can enumerate candidates against the salted hash. The salt stops rainbow tables; the HMAC key stops per-manifest enumeration. Even if the attacker finds the hash match, they can't forge the HMAC tag without the secret key. Tab E.3 has an animated simulator demonstrating this.
What are the conformance levels?
Level 1: Detect privacy modes, verify signatures, report "unsupported-mode" for unimplemented modes (never reject).
Level 2: Implement encryption and selective disclosure.
Level 3: Cross-mode chain walking, anonymous publication, attestation trust levels, metadata hardening.
The reference implementation achieves Level 3 (323 tests, 14 suites).
What about quantum computing?
Ed25519 and X25519 are not quantum-resistant. The key agreement field accepts alternative identifiers — HPKE is already supported. Post-quantum algorithms can be added via the same mechanism when standardized.

The Demo

What can I do in the demo?
Five tabs, zero network requests:

A (Keys): Generate identities, rotate keys, watch status transitions via Mock Clock.
B (Build): Sign JSON-LD, build version chains with deltas.
C (Query): SPARQL over verified content in Oxigraph WASM.
D (Resolve): Resolve URIs, prove byte-identical results, inject faults.
E (Privacy): All five modes interactive — encrypt, disclose, attest, go anonymous.
What are presets?
Pre-built scenarios that skip setup. Includes: simple verification, chain with deltas, key rotation, URDNA2015 + CIDv1, encrypted distribution, selective disclosure, anonymous whistleblower, attestation, and cross-mode privacy lifecycle chains.

Business Value

Why should organizations care?
Less infrastructure dependency. More data sovereignty. Cryptographic access control that travels with the data — not a policy that can be bypassed, but a mathematical property that makes unauthorized access computationally impossible.
Is this suitable for government systems?
Yes. The five modes map to government data patterns: proof of possession for classified existence claims, encrypted distribution for inter-agency sharing, selective disclosure for FOIA-like partial release, anonymous publication for whistleblower protection, attestation for cross-agency clearance verification. All local, no external dependencies.
Who is the privacy extension designed for?
Everyone. The Publisher Neutrality principle (§4.6) states that privacy modes serve non-human agents with equal rigor. The same encryption protecting medical records protects a synthetic agent's reasoning traces. The protocol makes no distinction about who deserves privacy.

HIRI is an open specification focused on clarity, determinism, and human-interpretable knowledge systems. Infrastructure is optional. Meaning is not.
