Frequently Asked Questions

HIRI (Hash-IRI) is a decentralized, edge-first, verifiable knowledge system. These questions address common technical and business concerns.

General Overview

What is HIRI?
HIRI is a decentralized semantic knowledge system where computation, reasoning, and learning occur at the edge. Knowledge is represented as JSON-LD artifacts that can be published, shared, reasoned over, or kept private — without requiring centralized infrastructure.
Is HIRI a product, a platform, or a protocol?
HIRI is primarily a specification and architectural model. Implementations can appear as desktop apps, browser applications, CLI tools, embedded systems, or server deployments, but none of those are required by the core spec.
How does HIRI differ from blockchain-based systems?
HIRI uses hash-linked manifest chains for sequential history, but requires no consensus protocol, no global state, and no mining. Each authority manages its own chain independently. Verification is local and deterministic — any node can verify any chain without contacting a network.

Architecture & Deployment

Does HIRI require servers or cloud infrastructure?
No. HIRI's canonical execution environment is a browser or a local Node.js runtime. Servers, clouds, and containers are optional optimizations, not architectural dependencies.
Can HIRI run on a desktop and a server?
Yes. The same specification supports both. A server deployment is simply a node that is always on and reachable. A desktop deployment is a node with local control and intermittent connectivity.
What is the "Edge-Canonical" principle?
It means the system must function fully without modification on a single machine. Anything that cannot do so is considered infrastructure, not core logic, and must be optional. The interactive demo runs entirely in your browser with zero network requests — proving this principle.
What storage backends does HIRI support?
HIRI is storage-agnostic. The kernel uses a StorageAdapter interface with get(hash) and put(hash, bytes) methods. The reference implementation includes InMemoryStorageAdapter, FileSystemAdapter (Node.js), and DelayedAdapter (for latency simulation). Any backend (IndexedDB, S3, IPFS) can implement this interface.

Privacy, Security, and Trust

Can a user run a HIRI node without exposing its contents?
Yes. A node may participate in a network while revealing only metadata, proofs, or capability claims — never raw knowledge artifacts unless explicitly published.
Can nodes collaborate without sharing private data?
Yes. HIRI supports participation via cryptographic commitments, hashes, summaries, or derived outputs rather than direct content sharing. Knowledge can remain local while its effects propagate.
How is hardware and local data protected?
HIRI does not require privileged access, background daemons, or system-level services. All execution occurs within the permissions of the host environment (browser sandbox or user-level Node process).
How does key rotation work?
Key rotation requires a dual-signature proof: both the old key and the new key sign a rotation claim. A configurable grace period allows manifests signed by the old key to remain valid for a defined duration after rotation. After the grace period expires, old-key signatures are rejected. This can be explored in the demo's Mock Clock slider.

Knowledge & Semantics

What role does JSON-LD play?
JSON-LD is the canonical representation for all knowledge, configuration, and inter-component contracts. This ensures transparency, inspectability, and long-term interoperability.
What is an A-Box in HIRI?
The A-Box represents asserted facts and observations. In HIRI, A-Boxes may be private, shared selectively, or published publicly as immutable knowledge artifacts.
What is entailment and why does HIRI default to "none"?
Entailment is the process of inferring new facts from existing ones (e.g., if SoftwareArchitect rdfs:subClassOf Person, then instances of SoftwareArchitect are also instances of Person). HIRI defaults to entailment mode "none" to ensure deterministic, auditable query results. Inference is a powerful tool, but it must be explicitly requested — not silently applied.
Is intermediate learning considered "published"?
Intermediate knowledge can be treated as first-class artifacts. This enables auditability, reproducibility, and collaborative reasoning without hidden model state.

Advanced Topics

Does the spec account for quantum encoding?
The spec is encoding-agnostic. While it does not mandate quantum representations, it does not preclude them. Any encoding that can be losslessly mapped to JSON-LD can participate.
How does HIRI handle offline or degraded operation?
Offline operation is a first-class mode. Partial information, deferred resolution, and explicit uncertainty are valid and expected states — not failures.
How does the delta mechanism work for version chains?
When content is updated, HIRI computes a JSON Patch (RFC 6902) delta between the previous and current versions. The delta is stored alongside the full content. During chain walking, the resolver first tries the delta path (faster, smaller). If the delta produces content whose hash doesn't match the manifest's content hash, it falls back to fetching full content. This makes deltas an optimization, not a trust boundary.

Business Value

Why should organizations care about HIRI?
HIRI reduces infrastructure dependency, increases data sovereignty, improves auditability, and enables collaboration without forced centralization — making it ideal for regulated, distributed, or mission-critical environments.
Is this suitable for government or critical systems?
Yes. The architecture aligns naturally with zero-trust principles, air-gapped operation, provenance tracking, and formal reasoning. It is designed to be safe to propose, evaluate, and pilot.

HIRI is an open specification focused on clarity, determinism, and human-interpretable knowledge systems. Infrastructure is optional. Meaning is not.
