Protocol Overview

HIRI (Hash-IRI) is a protocol for decentralized, verifiable knowledge that operates local-first and offline-capable. It separates data access from truth verification: anyone can host data, but only the authority's cryptographic signatures make it trustworthy.

The protocol operates through four layers:

LayerPurposeKey Concepts
1. FoundationIdentity & integrityHash-IRI identifiers, Resolution Manifests, Key Documents
2. ResolutionFetch & verifyManifest fetching, signature verification, chain walking
3. Storeless IndexRDF indexingDelta application, triple management, privacy-aware loading
4. QueryLocal queryingSPARQL execution, entailment control, result verification
Every operation in the protocol can be experienced in the interactive demo, which runs entirely in your browser with zero network requests.

Architecture

The codebase enforces a strict three-layer architecture:

Layer 0: src/kernel/ ← Pure computation. No I/O. No dependencies. Layer 1: src/composition/ ← Optional. Orchestration concepts. Layer 2: src/adapters/ ← Infrastructure: crypto, storage, RDF engines. Rules: • Layer 0 MUST NOT import from Layer 1 or Layer 2 • All crypto operations injected via CryptoProvider interface • Kernel code has zero non-deterministic API calls • Enforced by automated purity tests (81/81 passing)

The kernel contains all protocol logic as pure functions. No Date.now(), Math.random(), fetch(), or file system access. Timestamps, randomness, and I/O are always injected as parameters.

Step 1: Identity & Authority

Every HIRI identity starts with an Ed25519 keypair. The authority is derived from the public key — a deterministic, human-readable identifier that serves as the root of trust. 

import { generateKeypair } from "./adapters/crypto/ed25519.js";
import { deriveAuthorityAsync } from "./kernel/authority.js";
import { defaultCryptoProvider } from "./adapters/crypto/provider.js";

// Generate an Ed25519 keypair
const keypair = await generateKeypair("key-1");

// Derive authority: hash(publicKey) → base58 → truncate 20 chars
const authority = await deriveAuthorityAsync(
  keypair.publicKey,
  keypair.algorithm,
  defaultCryptoProvider,
);

// Result: "key:ed25519:FTVfKaT7wJHVywEPbGCG"
// URI:    "hiri://key:ed25519:FTVfKaT7wJHVywEPbGCG"
Authority derivation is deterministic: the same public key always produces the same authority string. This means anyone can verify that a URI belongs to a specific key by re-deriving the authority from the public key.

Step 2: Content & Signing

Content is signed through a pipeline: canonicalize, hash, build manifest, sign. The manifest is the trust anchor — it binds content to an identity with a cryptographic signature.

1
Canonicalize content via JCS (RFC 8785)

Deterministic JSON serialization ensures the same object always produces the same bytes.

2
Hash the canonical bytes (SHA-256)

Produces a content-addressed identifier: sha256:e08da327...

3
Build unsigned manifest

Embeds content hash, format, size, version, branch, and timing metadata.

4
Sign with Ed25519 key

The manifest body (minus signature) is canonicalized and signed. Signature is multibase-encoded (z + base58).

import { stableStringify } from "./kernel/canonicalize.js";
import { buildUnsignedManifest } from "./kernel/manifest.js";
import { signManifest } from "./kernel/signing.js";
import { hashManifest } from "./kernel/chain.js";

const crypto = defaultCryptoProvider;

// 1. Canonicalize & hash content
const content = { "@context": { "schema": "http://schema.org/" },
                  "@id": `hiri://${authority}/data/person`,
                  "@type": "schema:Person",
                  "schema:name": "Dana Reeves" };
const canonical = stableStringify(content);
const contentBytes = new TextEncoder().encode(canonical);
const contentHash = await crypto.hash(contentBytes);

// 2. Build unsigned manifest
const unsigned = buildUnsignedManifest({
  id: `hiri://${authority}/data/person`,
  version: 1,
  branch: "main",
  created: "2025-01-15T00:00:00Z",
  contentHash,
  contentFormat: "application/ld+json",
  contentSize: contentBytes.length,
  canonicalization: "JCS",
});

// 3. Sign
const signed = await signManifest(
  unsigned, keypair, "2025-01-15T00:00:00Z", crypto
);

// 4. Hash the signed manifest (for chain linking)
const manifestHash = await hashManifest(signed, crypto);

Step 3: Version Chains

When content is updated, the new manifest links to its predecessor via hiri:chain. This creates a tamper-evident history without any central authority or consensus protocol.

V1 (genesis) V2 V3 ┌──────────┐ ┌──────────┐ ┌──────────┐ │ version:1│←───│ version:2│←───│ version:3│ │ content:A│ │ content:B│ │ content:C│ │ no chain │ │ prev: V1 │ │ prev: V2 │ │ │ │ genesis:V1│ │ genesis:V1│ │ │ │ depth: 2 │ │ depth: 3 │ │ │ │ delta:A→B│ │ delta:B→C│ └──────────┘ └──────────┘ └──────────┘

Chain validation checks six rules at each link:

  1. ID consistency — same @id across all manifests
  2. Version monotonicity — each version > previous version
  3. Previous hash — chain.previous matches computed hash of predecessor
  4. Genesis immutability — chain.genesisHash is constant throughout the chain
  5. Depth accuracy — chain.depth increments correctly
  6. Branch consistency — chain.previousBranch matches predecessor's branch
import { validateChainLink, verifyChain } from "./kernel/chain.js";

// Validate a single link
const linkResult = await validateChainLink(v2Manifest, v1Manifest, crypto);
// { valid: true } or { valid: false, reason: "..." }

// Walk the full chain from head to genesis
const chainResult = await verifyChain(
  headManifest,
  publicKey,
  async (hash) => storage.get(hash),  // manifest fetcher
  async (hash) => storage.get(hash),  // content fetcher
  crypto,
);
// { valid: true, depth: 3, warnings: [], manifests: [...] }

Deltas

Each chain link can include a JSON Patch delta (RFC 6902) describing what changed. During chain walking, the resolver tries the delta path first. If the delta produces content whose hash doesn't match, it falls back to fetching full content. 

import { buildDelta, verifyDelta } from "./kernel/delta.js";
import { applyPatch } from "./kernel/json-patch.js";

// Build delta metadata
const { delta, serialized } = await buildDelta(
  patchOperations, previousContentHash, crypto
);

// Verify: apply patch, hash result, compare to expected
const result = await verifyDelta(
  delta, patchOperations, previousContentBytes,
  currentContentHash, crypto
);

Step 4: URI Resolution

The resolve() function is the core entry point for consumers. Given a HIRI URI, a storage adapter, and verification parameters, it returns verified content or throws a typed error.

1
Parse URI

hiri://key:ed25519:FTVf.../data/person → authority, type, identifier

2
Fetch & verify manifest

Retrieve manifest bytes by hash, verify computed hash matches, parse JSON.

3
Verify signature

Cryptographic Ed25519 verification against the authority's public key.

4
Walk chain (if chained)

Traverse chain.previous links back to genesis, verifying each link.

5
Fetch & verify content

Retrieve content bytes by hash, verify hash matches manifest's content.hash.

import { resolve, ResolutionError } from "./kernel/resolve.js";

try {
  const result = await resolve(
    "hiri://key:ed25519:FTVf.../data/person",
    storageAdapter,
    {
      crypto: defaultCryptoProvider,
      publicKey: keypair.publicKey,
      manifestHash: "sha256:4a457177...",
      // Optional: key lifecycle
      keyDocument: keyDoc,
      verificationTime: "2025-09-01T00:00:00Z",
    },
  );

  // result.content     — Uint8Array of verified bytes
  // result.contentHash — "sha256:..."
  // result.manifest    — the signed manifest
  // result.authority   — derived authority string
  // result.warnings    — key lifecycle warnings (if any)
} catch (e) {
  if (e instanceof ResolutionError) {
    // e.code: "STORAGE_CORRUPTION" | "SIGNATURE_VERIFICATION_FAILED"
    //       | "MANIFEST_NOT_FOUND" | "AUTHORITY_MISMATCH" | ...
  }
}

The resolver is storage-agnostic: it works identically with InMemoryStorageAdapter, FileSystemAdapter, or any custom adapter implementing get(hash) and put(hash, bytes). The demo's Tab D proves byte-identical results across adapters with different latencies.

Step 5: Key Lifecycle

HIRI supports key rotation and revocation through Key Documents. A Key Document declares which keys are active, rotated (with grace periods), or revoked (with retroactive invalidation).

Key States

StatusMeaningManifests Signed By This Key
activeKey is currently validAccepted
rotated-graceKey was rotated, within grace periodAccepted with warning
rotated-expiredKey was rotated, grace period expiredRejected
revokedKey was revoked (compromise suspected)Rejected after manifestsInvalidAfter
unknownKey not found in Key DocumentRejected

Rotation Proof

Key rotation requires a dual-signature proof: both the old key and the new key must sign a rotation claim. This prevents unauthorized rotation. 

import { resolveSigningKey, verifyRotationProof } from "./kernel/key-lifecycle.js";
import { buildKeyDocument } from "./kernel/manifest.js";
import { signKeyDocument } from "./kernel/signing.js";

// Resolve key status at a specific point in time
const result = resolveSigningKey(manifest, keyDocument, "2025-09-01T00:00:00Z");
// { keyStatus: "rotated-grace", valid: true, keyId: "key-1",
//   warning: "Signed by rotated key within grace period" }

// Verify rotation proof (dual signatures)
const proofValid = await verifyRotationProof(rotatedKey, keyDocument, crypto);
Try it live: The demo's Tab A has a Mock Clock slider that lets you drag through time and watch key statuses transition: active → rotated-grace → rotated-expired.

Step 6: RDF & SPARQL Querying

Verified content (JSON-LD) can be loaded into an in-memory RDF graph and queried with SPARQL. The graph engine runs locally — no database server required. 

import { OxigraphRDFStore } from "./adapters/rdf/oxigraph-store.js";

// Create an in-memory RDF store
const store = new OxigraphRDFStore();

// Load verified JSON-LD content
await store.load(contentBytes, "application/ld+json", `hiri://${authority}/`);

// Execute SPARQL
const result = await store.query(
  `SELECT ?name WHERE { ?s <http://schema.org/name> ?name }`,
  store,
);
// result.bindings = [{ name: { type: "literal", value: "Dana Reeves" } }]

Entailment Control

HIRI defaults to entailment mode "none": no RDFS/OWL inference is applied. This ensures deterministic, auditable query results. Even if a dataset declares rdfs:subClassOf relationships, they are not followed unless explicitly requested. 

// The entailment trap: SoftwareArchitect subClassOf Person
// Under mode="none":
// SELECT ?s WHERE { ?s a schema:Person }            → 0 results
// SELECT ?s WHERE { ?s a ex:SoftwareArchitect }     → 1 result

import { resolveEntailmentMode } from "./kernel/graph-builder.js";

const mode = resolveEntailmentMode(manifest["hiri:semantics"]);
// "none" (default) | "materialized" | "runtime"

API: Manifest Construction

FunctionSignatureDescription
buildUnsignedManifest (params: ManifestParams) → UnsignedManifest Build an unsigned ResolutionManifest. Pure, synchronous.
buildKeyDocument (params: KeyDocumentParams) → UnsignedKeyDocument Build an unsigned KeyDocument. Pure, synchronous.
prepareContent (jsonLd, placeholder, authority) → string Replace authority placeholder and canonicalize via JCS.

API: Signing & Verification

FunctionSignatureDescription
signManifest (unsigned, key, timestamp, crypto) → Promise<ResolutionManifest> Sign a manifest with Ed25519. Returns signed manifest with hiri:signature.
verifyManifest (manifest, publicKey, crypto) → Promise<boolean> Verify manifest signature against a public key.
signKeyDocument (unsigned, key, timestamp, crypto) → Promise<KeyDocument> Sign a Key Document.
verifyKeyDocument (doc, publicKey, crypto) → Promise<boolean> Verify Key Document signature.

API: Chain Validation

FunctionSignatureDescription
hashManifest (manifest, crypto) → Promise<string> SHA-256 hash of the complete signed manifest.
validateChainLink (current, previous, crypto) → Promise<ChainValidation> Validate a single chain link (6 rules).
verifyChain (head, publicKey, fetchManifest, fetchContent, crypto) → Promise<ChainWalkResult> Walk chain head-to-genesis, verifying signatures and links.
verifyChainWithKeyLifecycle (head, keyDoc, time, fetchManifest, fetchContent, crypto) → Promise<ChainWalkResult> Walk chain with key lifecycle awareness. No short-circuit on failure.

API: Resolution

FunctionSignatureDescription
resolve (uri, storage, options) → Promise<VerifiedContent> Full resolution pipeline: parse URI, verify manifest, walk chain, fetch content.

ResolveOptions:

  • crypto — CryptoProvider instance
  • publicKey — Uint8Array of the authority's public key
  • manifestHash — hash of the head manifest to resolve
  • keyDocument? — optional Key Document for lifecycle-aware verification
  • verificationTime? — ISO 8601 timestamp for key status evaluation

ResolutionError codes:

  • MANIFEST_NOT_FOUND — manifest hash not in storage
  • STORAGE_CORRUPTION — content hash mismatch
  • SIGNATURE_VERIFICATION_FAILED — signature doesn't match public key
  • AUTHORITY_MISMATCH — URI authority doesn't match derived authority
  • CHAIN_VERIFICATION_FAILED — chain walk found invalid link

API: Key Lifecycle

FunctionSignatureDescription
resolveSigningKey (manifest, keyDoc, verificationTime) → KeyVerificationResult Determine key status: active, rotated-grace, rotated-expired, revoked, unknown.
verifyManifestWithKeyLifecycle (manifest, keyDoc, time, crypto) → Promise<KeyVerificationResult> Resolve key status + cryptographic signature verification.
verifyRotationProof (rotatedKey, keyDoc, crypto) → Promise<boolean> Verify dual-signature rotation proof (old-key + new-key).

API: Query Execution

FunctionSignatureDescription
buildGraph (content, format, baseURI, config, indexFactory) → Promise<RDFIndex> Load verified content into an RDF index with entailment control.
resolveEntailmentMode (semantics?) → EntailmentMode Extract entailment mode from manifest semantics. Default: "none".
executeQuery (sparql, index, engine) → Promise<QueryResult> Execute SPARQL query against an RDF index.

Design Constraints

These are enforced by automated tests and the purity checker:

  • No non-deterministic APIs in kernel: no Date.now(), Math.random(), crypto.getRandomValues(), process.env, fetch, XMLHttpRequest
  • No kernel → adapter imports: kernel code never imports from src/adapters/
  • Deterministic serialization: all hashing uses JCS canonicalization via stableStringify()
  • Multibase encoding: signatures and keys use z prefix + base58
  • Timestamps as parameters: all temporal operations accept ISO 8601 strings, never call the system clock
  • Zero runtime dependencies: only devDependencies (TypeScript, test runner, esbuild)
Entailment modes "materialized" and "runtime" are not yet implemented. buildGraph() will throw if either mode is requested. Only "none" is available in the current version. This is deliberate — inference must be explicitly implemented and tested before being made available.
