Protocol Overview

HIRI (Hash-IRI) is a protocol for decentralized, verifiable knowledge that operates local-first and offline-capable. It separates data access from truth verification: anyone can host data, but only the authority's cryptographic signatures make it trustworthy.

The protocol operates through four layers:

LayerPurposeKey Concepts
1. FoundationIdentity & integrityHash-IRI identifiers, Resolution Manifests, Key Documents
2. ResolutionFetch & verifyManifest fetching, signature verification, chain walking
3. Storeless IndexRDF indexingDelta application, triple management, privacy-aware loading
4. QueryLocal queryingSPARQL execution, entailment control, result verification
Every operation in the protocol can be experienced in the interactive demo, which runs entirely in your browser with zero network requests.

Architecture

The codebase enforces a strict three-layer architecture:

Layer 0: src/kernel/ ← Pure computation. No I/O. No dependencies. Layer 1: src/composition/ ← Optional. Orchestration concepts. Layer 2: src/adapters/ ← Infrastructure: crypto, storage, RDF engines. Rules: • Layer 0 MUST NOT import from Layer 1 or Layer 2 • All crypto operations injected via CryptoProvider interface • Kernel code has zero non-deterministic API calls • Enforced by automated purity tests (323/323 passing)

The kernel contains all protocol logic as pure functions. No Date.now(), Math.random(), fetch(), or file system access. Timestamps, randomness, and I/O are always injected as parameters.

Step 1: Identity & Authority

Every HIRI identity starts with an Ed25519 keypair. The authority is derived from the public key — a deterministic, human-readable identifier that serves as the root of trust. 

import { generateKeypair } from "./adapters/crypto/ed25519.js";
import { deriveAuthority } from "./kernel/authority.js";

// Generate an Ed25519 keypair
const keypair = await generateKeypair("key-1");

// Derive authority: "z" + base58btc(publicKey) — no hashing, no truncation
const authority = deriveAuthority(keypair.publicKey, keypair.algorithm);

// Result: "key:ed25519:z6MkhaXgBZDvotDkL5257faiztiGiC2QtKLGpbnnEGta2doK"
// URI:    "hiri://key:ed25519:z6MkhaXgBZDvotDkL5257faiztiGiC2QtKLGpbnnEGta2doK"
Authority derivation is deterministic: the same public key always produces the same authority string. This means anyone can verify that a URI belongs to a specific key by re-deriving the authority from the public key.

Step 2: Content & Signing

Content is signed through a pipeline: canonicalize, hash, build manifest, sign. The manifest is the trust anchor — it binds content to an identity with a cryptographic signature.

1
Canonicalize content via JCS (RFC 8785)

Deterministic JSON serialization ensures the same object always produces the same bytes.

2
Hash the canonical bytes (SHA-256)

Produces a content-addressed identifier: sha256:e08da327...

3
Build unsigned manifest

Embeds content hash, format, size, version, branch, and timing metadata.

4
Sign with Ed25519 key

The manifest body (minus signature) is canonicalized and signed. Signature is multibase-encoded (z + base58).

import { stableStringify } from "./kernel/canonicalize.js";
import { buildUnsignedManifest } from "./kernel/manifest.js";
import { signManifest } from "./kernel/signing.js";
import { hashManifest } from "./kernel/chain.js";

const crypto = defaultCryptoProvider;

// 1. Canonicalize & hash content
const content = { "@context": { "schema": "http://schema.org/" },
                  "@id": `hiri://${authority}/data/person`,
                  "@type": "schema:Person",
                  "schema:name": "Dana Reeves" };
const canonical = stableStringify(content);
const contentBytes = new TextEncoder().encode(canonical);
const contentHash = await crypto.hash(contentBytes);

// 2. Build unsigned manifest
const unsigned = buildUnsignedManifest({
  id: `hiri://${authority}/data/person`,
  version: "1",
  branch: "main",
  created: "2025-01-15T00:00:00Z",
  contentHash,
  addressing: "raw-sha256",
  contentFormat: "application/ld+json",
  contentSize: contentBytes.length,
  canonicalization: "JCS",
});

// 3. Sign (profile must match content.canonicalization)
const signed = await signManifest(
  unsigned, keypair, "2025-01-15T00:00:00Z", "JCS", crypto
);

// 4. Hash the signed manifest (for chain linking)
const manifestHash = await hashManifest(signed, crypto);

Step 3: Version Chains

When content is updated, the new manifest links to its predecessor via hiri:chain. This creates a tamper-evident history without any central authority or consensus protocol.

V1 (genesis) V2 V3 ┌──────────┐ ┌──────────┐ ┌──────────┐ │ version:1│←───│ version:2│←───│ version:3│ │ content:A│ │ content:B│ │ content:C│ │ no chain │ │ prev: V1 │ │ prev: V2 │ │ │ │ genesis:V1│ │ genesis:V1│ │ │ │ depth: 2 │ │ depth: 3 │ │ │ │ delta:A→B│ │ delta:B→C│ └──────────┘ └──────────┘ └──────────┘

Chain validation checks six rules at each link:

  1. ID consistency — same @id across all manifests
  2. Version monotonicity — each version > previous version
  3. Previous hash — chain.previous matches computed hash of predecessor
  4. Genesis immutability — chain.genesisHash is constant throughout the chain
  5. Depth accuracy — chain.depth increments correctly
  6. Branch consistency — chain.previousBranch matches predecessor's branch
import { validateChainLink, verifyChain } from "./kernel/chain.js";

// Validate a single link
const linkResult = await validateChainLink(v2Manifest, v1Manifest, crypto);
// { valid: true } or { valid: false, reason: "..." }

// Walk the full chain from head to genesis
const chainResult = await verifyChain(
  headManifest,
  publicKey,
  async (hash) => storage.get(hash),  // manifest fetcher
  async (hash) => storage.get(hash),  // content fetcher
  crypto,
);
// { valid: true, depth: 3, warnings: [], manifests: [...] }

Deltas

Each chain link can include a delta describing what changed. JCS-profile manifests use JSON Patch (RFC 6902); URDNA2015-profile manifests use RDF Patch (triple-level add/remove). During chain walking, the resolver tries the delta path first. If the delta produces content whose hash doesn't match, it falls back to fetching full content. 

import { buildDelta, verifyDelta } from "./kernel/delta.js";
import { applyPatch } from "./kernel/json-patch.js";

// Build delta metadata
const { delta, serialized } = await buildDelta(
  patchOperations, previousContentHash, crypto
);

// Verify: apply patch, hash result, compare to expected
const result = await verifyDelta(
  delta, patchOperations, previousContentBytes,
  currentContentHash, "JCS", crypto
);

Step 4: URI Resolution

The resolve() function is the core entry point for consumers. Given a HIRI URI, a storage adapter, and verification parameters, it returns verified content or throws a typed error.

1
Parse URI

hiri://key:ed25519:FTVf.../data/person → authority, type, identifier

2
Fetch & verify manifest

Retrieve manifest bytes by hash, verify computed hash matches, parse JSON.

3
Verify signature

Cryptographic Ed25519 verification against the authority's public key.

4
Walk chain (if chained)

Traverse chain.previous links back to genesis, verifying each link.

5
Fetch & verify content

Retrieve content bytes by hash, verify hash matches manifest's content.hash.

import { resolve, ResolutionError } from "./kernel/resolve.js";

try {
  const result = await resolve(
    "hiri://key:ed25519:FTVf.../data/person",
    storageAdapter,
    {
      crypto: defaultCryptoProvider,
      publicKey: keypair.publicKey,
      manifestHash: "sha256:4a457177...",
      // Optional: key lifecycle
      keyDocument: keyDoc,
      verificationTime: "2025-09-01T00:00:00Z",
    },
  );

  // result.content     — Uint8Array of verified bytes
  // result.contentHash — "sha256:..."
  // result.manifest    — the signed manifest
  // result.authority   — derived authority string
  // result.warnings    — key lifecycle warnings (if any)
} catch (e) {
  if (e instanceof ResolutionError) {
    // e.code: "STORAGE_CORRUPTION" | "SIGNATURE_VERIFICATION_FAILED"
    //       | "MANIFEST_NOT_FOUND" | "AUTHORITY_MISMATCH" | ...
  }
}

The resolver is storage-agnostic: it works identically with InMemoryStorageAdapter, FileSystemAdapter, or any custom adapter implementing get(hash) and put(hash, bytes). The demo's Tab D proves byte-identical results across adapters with different latencies.

Step 5: Key Lifecycle

HIRI supports key rotation and revocation through Key Documents. A Key Document declares which keys are active, rotated (with grace periods), or revoked (with retroactive invalidation).

Key States

StatusMeaningManifests Signed By This Key
activeKey is currently validAccepted
rotated-graceKey was rotated, within grace periodAccepted with warning
rotated-expiredKey was rotated, grace period expiredRejected
revokedKey was revoked (compromise suspected)Rejected after manifestsInvalidAfter
unknownKey not found in Key DocumentRejected

Rotation Proof

Key rotation requires a dual-signature proof: both the old key and the new key must sign a rotation claim. This prevents unauthorized rotation. 

import { resolveSigningKey, verifyRotationProof } from "./kernel/key-lifecycle.js";
import { buildKeyDocument } from "./kernel/manifest.js";
import { signKeyDocument } from "./kernel/signing.js";

// Resolve key status at a specific point in time
const result = resolveSigningKey(manifest, keyDocument, "2025-09-01T00:00:00Z");
// { keyStatus: "rotated-grace", valid: true, keyId: "key-1",
//   warning: "Signed by rotated key within grace period" }

// Verify rotation proof (dual signatures)
const proofValid = await verifyRotationProof(rotatedKey, keyDocument, crypto);
Try it live: The demo's Tab A has a Mock Clock slider that lets you drag through time and watch key statuses transition: active → rotated-grace → rotated-expired.

Step 6: RDF & SPARQL Querying

Verified content (JSON-LD) can be loaded into an in-memory RDF graph and queried with SPARQL. The graph engine runs locally — no database server required. 

import { OxigraphRDFStore } from "./adapters/rdf/oxigraph-store.js";

// Create an in-memory RDF store
const store = new OxigraphRDFStore();

// Load verified JSON-LD content
await store.load(contentBytes, "application/ld+json", `hiri://${authority}/`);

// Execute SPARQL
const result = await store.query(
  `SELECT ?name WHERE { ?s <http://schema.org/name> ?name }`,
  store,
);
// result.bindings = [{ name: { type: "literal", value: "Dana Reeves" } }]

Entailment Control

HIRI defaults to entailment mode "none": no RDFS/OWL inference is applied. This ensures deterministic, auditable query results. Even if a dataset declares rdfs:subClassOf relationships, they are not followed unless explicitly requested. 

// The entailment trap: SoftwareArchitect subClassOf Person
// Under mode="none":
// SELECT ?s WHERE { ?s a schema:Person }            → 0 results
// SELECT ?s WHERE { ?s a ex:SoftwareArchitect }     → 1 result

import { resolveEntailmentMode } from "./kernel/graph-builder.js";

const mode = resolveEntailmentMode(manifest["hiri:semantics"]);
// "none" (default) | "materialized" | "runtime"

Privacy Extension: Overview

The privacy extension adds five modes that control content visibility and identity linkability. Every privacy-aware resolution goes through a single entry point: 

import { resolveWithPrivacy } from "hiri/privacy/resolve.js";

const result = await resolveWithPrivacy(uri, storage, {
  crypto,
  publicKey,
  manifestHash,
  // Optional, mode-dependent:
  decryptionKey,       // X25519 private key (Mode 2, Mode 3)
  recipientId,         // Your recipient identifier (Mode 2, Mode 3)
  subjectPublicKey,    // Subject authority's key (Mode 5)
  attestorKeyStatus,   // "active" | "revoked" (Mode 5)
  verificationTime,    // ISO 8601 timestamp for staleness checks
});

The result always contains:

FieldTypeDescription
verifiedbooleanSignature + chain integrity
privacyModestring"public" | "proof-of-possession" | "encrypted" | "selective-disclosure" | "anonymous" | "attestation"
contentStatusstringMode-specific content verification outcome
warningsstring[]Staleness, key status, verification notes. The staleness check uses strict greater-than comparison — a KeyDocument whose age exactly equals keyDocumentMaxAge is not stale.
identityType?stringFor anonymous: "anonymous-ephemeral" | "pseudonymous"
decryptedContent?Uint8ArrayMode 2 with valid key
disclosedNQuads?string[]Mode 3 mandatory statements
attestationResult?objectMode 5 trust level and claim
Single entry point: A developer who reads only the resolveWithPrivacy() signature and result type can integrate privacy-aware resolution without understanding the internals. Every other section below is "how to build the manifests that resolver consumes."

Mode Decision Tree

Does the content need to exist publicly? ├── Yes → Mode: public (no privacy block needed) └── No ├── Prove you HAVE the data without revealing it? │ └── Yes → Mode 1: Proof of Possession (§6) │ ├── Specific recipients read the full content? │ └── Yes → Mode 2: Encrypted Distribution (§7) │ ├── Recipients see SOME statements but not others? │ └── Yes → Mode 3: Selective Disclosure (§8) │ ├── Publisher's identity hidden? │ └── Yes → Mode 4: Anonymous Publication (§9) │ └── Combined with any content visibility │ └── Third party vouches for a property? └── Yes → Mode 5: Third-Party Attestation (§10)

Modes 1–3 control content visibility. Mode 4 controls identity visibility. Mode 5 enables cross-authority trust. Modes compose: an anonymous publisher (Mode 4) can use selective disclosure (Mode 3) for content — the "anonymous whistleblower" use case (§4.6).

Encrypted Distribution (Mode 2)

The Dual Hash Model

Encrypted manifests have two hashes:

manifest["hiri:content"].hash → ciphertext hash (public) manifest["hiri:privacy"].parameters.plaintextHash → plaintext hash (in privacy block)

Storage adapters use the ciphertext hash for content-addressed lookup. The plaintext hash is what getLogicalPlaintextHash() returns for chain walking. 

import { encryptContent } from "hiri/privacy/encryption.js";
import { buildEncryptedManifest } from "hiri/privacy/encrypted-manifest.js";

// Encrypt for multiple recipients
const encResult = await encryptContent(
  plaintextBytes,
  new Map([["alice", alice.publicKey], ["bob", bob.publicKey]]),
  crypto,
);

// Build the manifest (content.hash = ciphertext hash)
const manifest = buildEncryptedManifest({
  baseManifestParams: { id, version, branch, created, addressing, canonicalization },
  encryptionResult: encResult,
  plaintextFormat: "application/ld+json",
  plaintextSize: plaintextBytes.length,
});

Three possible resolution outcomes:

ScenariocontentStatusverified
No decryption key provided"ciphertext-verified"true
Valid recipient key"decrypted-verified"true
Wrong key / not a recipient"decryption-failed"true
verified is true in ALL cases. Signature verification and content decryption are orthogonal. A manifest with a valid signature but a failed decryption is still a verified manifest — you just can't read the content.

HKDF Domain Separation

Mode 2 uses HKDF label "hiri-cek-v1.1" for CEK distribution. Mode 3 uses "hiri-hmac-v1.1" for HMAC key distribution. These labels MUST differ — same ECDH shared secret, different derived keys. Wrong label = AES-GCM authentication failure.

Selective Disclosure (Mode 3)

Selective disclosure canonicalizes JSON-LD into individual N-Quad statements, then discloses some publicly while withholding others. Each statement gets a salted SHA-256 hash (in the statement index) and an HMAC-SHA-256 tag (for authorized verification). 

import { buildStatementIndex } from "hiri/privacy/statement-index.js";
import { generateHmacTags, encryptHmacKeyForRecipients } from "hiri/privacy/hmac-disclosure.js";

// Build index from canonical N-Quads
const indexResult = await buildStatementIndex(canonicalNQuads);

// Generate HMAC tags
const hmacKey = crypto.getRandomValues(new Uint8Array(32));
const tags = generateHmacTags(statements, hmacKey, indexResult.indexSalt);

// Encrypt HMAC key for recipients (auto-generates ephemeral keypair + IV)
const distribution = await encryptHmacKeyForRecipients(
  hmacKey, recipientPublicKeys, disclosureMap,
);
hmacKey.fill(0);

The Byte-Level Concatenation Rule (CRITICAL)

This is the #1 interoperability failure mode. The wrong approach produces valid SHA-256 hashes that pass all local tests but fail against any other implementation.

The spec (§8.4.2) requires:

statementHash[i] = SHA-256(concat(rawSaltBytes, UTF-8(statement)))

This is byte-level concatenation of two Uint8Array values. NOT string concatenation.

// CORRECT: byte-level concat
const stmtBytes = new TextEncoder().encode(statement);
const input = new Uint8Array(salt.length + stmtBytes.length);
input.set(salt, 0);
input.set(stmtBytes, salt.length);
const hash = await crypto.subtle.digest("SHA-256", input);

// WRONG: string concat (produces different hash!)
const saltB64 = base64urlEncode(salt);
const input = new TextEncoder().encode(saltB64 + statement);  // ← DIFFERENT BYTES
const hash = await crypto.subtle.digest("SHA-256", input);

HMAC Verification Boundary

The published SD content blob only contains mandatory N-Quad strings. Non-mandatory statement text is not stored publicly. This creates a verification boundary:

  • Mandatory statements: Verifiable by anyone. Text is in the published blob.
  • Non-mandatory disclosed statements: Authorized by HMAC key decryption, but tag verification requires out-of-band statement delivery.
// CORRECT: only verify statements whose text is available
for (const idx of disclosedIndices) {
  const mandatoryPos = mandatoryStatements.indexOf(idx);
  if (mandatoryPos !== -1) {
    // Text available — verify HMAC tag
    verifyHmacTag(mandatoryNQuads[mandatoryPos], hmacKey, salt, tags[idx]);
  }
  // Non-mandatory: HMAC key decryption proves authorization.
  // Tag verification requires out-of-band statement delivery.
}

Serialization at the JSON Boundary

encryptHmacKeyForRecipients() returns Uint8Array fields (ephemeralPublicKey, iv, encryptedHmacKey). The manifest's privacy block is JSON-serialized for storage. Uint8Array serializes as {"0":1,"1":2,...} — not a hex string. Convert all binary fields to hex before embedding: 

// WRONG: raw Uint8Array won't survive JSON round-trip
parameters.hmacKeyRecipients = distribution;

// CORRECT: convert to hex strings matching HmacKeyRecipients interface
parameters.hmacKeyRecipients = {
  ephemeralPublicKey: bytesToHex(distribution.ephemeralPublicKey),
  iv: bytesToHex(distribution.iv),
  keyAgreement: "X25519-HKDF-SHA256",
  recipients: distribution.recipients.map(r => ({
    id: r.id,
    encryptedHmacKey: bytesToHex(r.encryptedHmacKey),
    disclosedStatements: r.disclosedStatements,
  })),
};

Anonymous Publication (Mode 4)

Anonymous publication separates identity from content. Two authority types:

TypeLinkabilityKey LifecyclePrivate Key
EphemeralComputationally unlinkableNo KeyDocument (§9.5)Destroyed after signing
PseudonymousLinkable (same authority)Optional KeyDocumentRetained for reuse
import { generateEphemeralAuthority } from "hiri/privacy/anonymous.js";

const eph = await generateEphemeralAuthority();
const signed = await signManifest(manifest, ephSigningKey, created, "JCS", crypto);

// MUST destroy the private key after signing
eph.privateKey.fill(0);

Anonymous publication combines with any content visibility mode via the contentVisibility parameter: "public", "encrypted", "selective-disclosure", or "private" (PoP).

Third-Party Attestation (Mode 5)

Attestation lets an examiner (attestor) vouch for a property of content held by another authority (subject). The attestation IS the manifest — there is no separate content block. 

import { buildAttestationManifest, verifyAttestation } from "hiri/privacy/attestation.js";

const attestation = buildAttestationManifest({
  attestorAuthority,
  attestationId: "clearance-check-001",
  subject: { authority: subjectAuthority, manifestHash, contentHash },
  claim: { property: "security-clearance-valid", value: true, scope: "TS/SCI" },
  evidence: { method: "direct-examination", description: "..." },
});
// attestation has NO hiri:content block (§10.4)

Trust Levels

ScenariotrustLevelMeaning
Both signatures verified, attestor key active"full"Maximum assurance
Attestor verified, subject unavailable"partial"Trust the attestor's word
Attestor key revoked AND subject unavailable"unverifiable"No independent verification

Trust level is informational — the resolver reports it, the application decides what to do with it. Stale attestations (past validUntil) are not invalid; the claim was true when made.

Cross-Mode Chains (Level 3)

Content can transition between privacy modes across versions. Transitions MUST be monotonically decreasing in privacy:

proof-of-possession → encrypted ✓ (sharing with recipients) proof-of-possession → public ✓ (full publication) encrypted → public ✓ (declassification) encrypted → proof-of-possession ✗ (cannot withdraw access) public → anything private ✗ (cannot unpublish)

Raw content.hash comparison across a PoP → Encrypted transition FAILS because V1's hash is the plaintext hash and V2's hash is the ciphertext hash. Use getLogicalPlaintextHash() for all cross-version content comparisons:

V1: Proof of Possession content.hash = sha256:aaa getLogicalPlaintextHash() → sha256:aaa V2: Encrypted Distribution content.hash = sha256:bbb getLogicalPlaintextHash() → sha256:aaa V3: Public content.hash = sha256:aaa getLogicalPlaintextHash() → sha256:aaa Raw content.hash V1→V2: aaa ≠ bbb FAILS Logical hash V1→V2: aaa = aaa SUCCEEDS

Common Privacy Mistakes

Ordered by frequency, not by spec section. These are the bugs every implementer hits.

#1 — String Concatenation Instead of Byte Concatenation

Symptom: Every salted hash and index root is wrong. Cross-implementation verification fails.
Cause: Using base64url(salt) + statement as a string instead of concat(rawSaltBytes, UTF-8(statement)) as byte arrays.
Fix: See the byte-level concatenation rule above. Test with vector B.13.

#2 — Uint8Array Fields in JSON-Serialized Manifests

Symptom: hexToBytes() receives [object Object]. X25519 ECDH gets length=0.
Cause: Placing raw Uint8Array from encryption functions directly into the manifest. After JSON round-trip, they become plain objects, not hex strings.
Fix: Hex-encode all binary fields before embedding in hiri:privacy.parameters. See the serialization section above.

#3 — Wrong HKDF Label

Symptom: AES-GCM authentication failure when decrypting a CEK or HMAC key.
Cause: Using "hiri-cek-v1.1" for Mode 3 or "hiri-hmac-v1.1" for Mode 2.
Fix: Mode 2 = "hiri-cek-v1.1". Mode 3 = "hiri-hmac-v1.1".

#4 — Passing Publisher Key as IV

Symptom: AES-GCM IV must be 12 bytes, got 32.
Cause: Passing a 32-byte X25519 public key as the optional iv parameter of encryptHmacKeyForRecipients().
Fix: Omit the 4th argument. The function generates its own ephemeral keypair and 12-byte IV internally.

#5 — HMAC Verification of Non-Mandatory Statements

Symptom: False HMAC tag failures for disclosed but non-mandatory statements.
Cause: Looking up statement text in mandatoryNQuads using indexOf(idx) which returns -1 for non-mandatory indices, producing an empty string.
Fix: Only verify HMAC tags for mandatory statements (where text is available). See the HMAC verification boundary section above.

#6 — Comparing Raw content.hash Across Modes

Symptom: Chain verification fails at a PoP → Encrypted transition.
Cause: Comparing content.hash directly instead of using getLogicalPlaintextHash().
Fix: Use getLogicalPlaintextHash() for all cross-version content comparisons.

#7 — Treating verified:true + decryption-failed as Error

Symptom: Application rejects a valid manifest because content couldn't be decrypted.
Cause: Conflating signature verification with content access. They are orthogonal.
Fix: verified: true = authentic manifest. contentStatus: "decryption-failed" = you don't have the right key.

Design Constraints

These are enforced by automated tests and the purity checker:

  • No non-deterministic APIs in kernel: no Date.now(), Math.random(), crypto.getRandomValues(), process.env, fetch, XMLHttpRequest
  • No kernel → adapter imports: kernel code never imports from src/adapters/
  • Deterministic serialization: JCS via stableStringify() or URDNA2015 via injected Canonicalizer
  • Multibase encoding: signatures and keys use z prefix + base58
  • Timestamps as parameters: all temporal operations accept ISO 8601 strings, never call the system clock
  • Minimal runtime dependencies: @noble/ed25519, jsonld, oxigraph, multiformats, @ipld/dag-cbor
Entailment modes "materialized" and "runtime" are not yet implemented. buildGraph() will throw if either mode is requested. Only "none" is available in the current version. This is deliberate — inference must be explicitly implemented and tested before being made available.
