Enterprise Deployment Guide

Deployment patterns, key management, regulatory compliance, and security architecture for organizations adopting the HIRI protocol.

1. Deployment Profiles

The protocol defines three deployment profiles (Privacy Extension §18.3). Choose based on your data sensitivity and regulatory environment.

Minimal Low Risk

Privacy Level 1 + Proof of Possession. For organizations needing provenance and integrity without confidentiality. Software bill of materials (SBOM), scientific data provenance, policy integrity verification.

Standard Regulated

Privacy Level 2 + Encrypted Distribution + Selective Disclosure. HIRI KeyDocument bootstrapping. Recommended metadata-hardening defaults. Suitable for general enterprise data, inter-organization sharing, and most regulatory environments.

High-Assurance Sensitive

Privacy Level 3 + all five modes + metadata hardening + transparency log checkpoints. For regulated data (HIPAA, financial), intelligence, legally privileged information, and environments requiring formal auditability of every operation.

2. Key Management

Key protection is outside the protocol's scope — this is true of every signing protocol (X.509, PGP, FIDO2, W3C Verifiable Credentials, JWTs). The protocol provides the mechanisms; the deployment provides the protection.

CryptoProvider Injection

All cryptographic operations flow through an injectable CryptoProvider interface. The reference implementation uses Web Crypto API, but enterprise deployments can substitute any backend: 

// Reference: Web Crypto (browser/Node.js)
const crypto = { sign, verify, hash, generateKeypair, ... };

// Enterprise: HSM-backed provider
const crypto = createHsmCryptoProvider({
  slot: 0,
  pin: process.env.HSM_PIN,
  // Keys never leave the HSM boundary
});

// All protocol operations accept the provider
const signed = await signManifest(unsigned, key, timestamp, profile, crypto);

The signing key never needs to exist in application memory. An HSM or secure enclave holds the private key and performs the Ed25519 signature internally. The protocol receives the signature bytes — it never sees the key.

When the cached KeyDocument exceeds keyDocumentMaxAge, the resolver emits a keyDocumentStale: true warning. The application layer decides whether to proceed or require a refresh. This is the same trade-off as CRL/OCSP caching in X.509: you can have offline verification or real-time revocation status, not both simultaneously.

3. Data Lifecycle & Erasure

HIRI artifacts are not “immutable once distributed.” The protocol is storage-agnostic — it defines get(hash) and has(hash), not distribute() or replicate(). The deployer controls every byte.

Correction: Version Chains

When data is wrong, publish a new version. V2 supersedes V1. The chain records the correction history — this is auditable correction, not absence of correction. Every database with an audit log works this way.

Erasure: Three Mechanisms

1. Storage-Level Deletion

Delete content bytes from the storage adapter. The manifest chain records that a version existed, but the content is no longer retrievable. Resolution returns “content not found.”

2. Cryptographic Erasure (Mode 2)

Destroy the content encryption key. The ciphertext remains but is permanently unreadable — computationally equivalent to deletion. This is a recognized GDPR-compliant erasure technique (acknowledged by UK ICO and French CNIL).

3. Retention Schedule via Storage Adapter TTLs

Implement time-to-live in the storage adapter. Content older than the retention policy is automatically purged. The manifest chain provides the audit trail that the data once existed and was removed per policy.

Privacy Monotonicity

The rule “once public, cannot be made private” is an integrity property: it prevents a publisher from claiming they never published something they did publish. It does not prevent deleting the content from storage. The chain records that V3 was public; if V3’s content bytes are deleted, resolution returns “content not found” — which is the correct behavior for erasure.

This is actually better for compliance than most systems: you can prove you responded to an erasure request (the chain shows the version existed and was then made unavailable), while actually erasing the content. Most databases cannot prove they deleted something without maintaining a log of the deletion — which itself contains metadata about the deleted record.

4. Regulatory Compliance

HIRI provides the cryptographic primitives. The deployment provides the governance. The protocol gives the controller the ability to locate, modify, restrict, and delete data:

  • Locate: Content-addressed hashes are globally unique identifiers. A storage adapter with an index locates every artifact by hash.
  • Modify: Version chains with deltas are the correction mechanism.
  • Restrict: Mode 1 (PoP) withholds entirely. Mode 2 (Encrypted) restricts to named recipients. Mode 3 (SD) restricts at the statement level.
  • Delete: Storage-level deletion removes content bytes. Cryptographic erasure (Mode 2 key destruction) makes content permanently unreadable.

GDPR (Arts. 16, 17, 25)

Right to rectification (Art. 16): Version chains. V2 corrects V1.

Right to erasure (Art. 17): Storage deletion + cryptographic erasure.

Privacy by design (Art. 25): The privacy extension — five modes, selective disclosure, encrypted distribution — is privacy by design.

CCPA / CPRA

Same mechanisms as GDPR. Controller deletes from storage. Cryptographic erasure for encrypted content. The content-addressed model provides the audit trail for “right to know” requests.

HIPAA

Access logging: The storage adapter can log every get() call. This is a deployment concern, not a protocol limitation.

Minimum necessary: Mode 3 selective disclosure shares exactly the fields needed and nothing more — statement-level access control maps directly to the minimum necessary standard.

PCI-DSS

Cardholder data should not be placed in HIRI artifacts. Use HIRI for payment metadata, attestations, and provenance — not card numbers or CVVs. Exclusion

ITAR / EAR

Geographic access control requires infrastructure (geo-fencing, network controls). This is not a HIRI-specific limitation — any data that leaves a controlled network is subject to this concern. Deploy HIRI within controlled perimeters for export-controlled data. Exclusion

5. Attestation & Regulated Data

Deployment Requirement: Attestation manifests referencing regulated data MUST use content hashes as subject references, MUST NOT contain directly or indirectly identifying information, and MUST rely on the originating system’s internal crosswalk for hash-to-person resolution.

A content hash is opaque outside the originating system. It is not personal data under GDPR, PHI under HIPAA, or personal information under CCPA — unless the attestation itself contains linkable identifiers. The constraint above ensures it does not.

Compliance Pattern

// Attestation references a content hash — not a person
{
  "hiri:attestation": {
    "subject": "hiri://sha256:a1b2c3...",   // opaque hash
    "claim": "clearance-verified",
    "validUntil": "2026-12-31T00:00:00Z"
  }
}

// The originating system maintains the crosswalk:
//   sha256:a1b2c3... → Employee #4821 (internal only)
// The attestation chain never sees the employee ID.

This separation means the attestation can be shared, cached, and verified by third parties without exposing the identity of the data subject. The originating system resolves the hash to a person only when authorized — and that resolution is a standard database lookup under existing access controls.

6. Security Architecture

Trust Bootstrapping (TOFU)

First contact with an authority has no out-of-band trust anchor. This is the Trust On First Use problem, shared by SSH, Signal, PGP, and every system without a centralized certificate authority. Recommended bootstrapping patterns:

  • DNS TXT records: Publish the authority string in a DNS TXT record for the organization’s domain. Verifiers resolve the domain to confirm the authority.
  • QR code exchange: Encode the authority in a QR code for in-person or out-of-band verification.
  • Organizational key registries: Maintain an internal directory mapping departments or roles to HIRI authorities.
  • Attestation chains: A trusted attestor (e.g., an organizational root) vouches for subordinate authorities via Mode 5 attestation manifests.

Artifact Integrity at Rest

Content-addressed storage with hash verification is the protection. If an attacker replaces an artifact on disk, the hash does not match, and verification fails. The resolver explicitly checks computedHash !== expectedHash and rejects on mismatch. This is the entire point of content-addressed storage.

Ephemeral Key Destruction

The reference implementation calls Uint8Array.fill(0) to overwrite ephemeral keys after use. This overwrites the JavaScript heap allocation, but the runtime, OS, and hardware may retain copies (swap files, SSD wear leveling, memory dumps). True ephemeral key destruction requires hardware support — secure enclaves, TPMs with key attestation. For deployments where unlinkability is a hard requirement, use hardware-backed key generation via the CryptoProvider injection point.

Selective Disclosure: Field Entropy

Low-entropy fields (booleans, small enumerations like blood types, date ranges) are vulnerable to dictionary enumeration even with salted hashing. The salt defeats rainbow tables; the HMAC key defeats per-manifest enumeration. But for fields with very small value spaces:

  • Combine fields: Disclose low-entropy fields together with high-entropy fields in a single composite statement.
  • Pad values: Use metadata hardening (Privacy Extension §15.3) to pad statement counts and obscure field boundaries.
  • Withhold entirely: If a field’s value space is small enough to enumerate (e.g., boolean, 8 blood types), consider Mode 1 (PoP) for that portion of the data instead of selective disclosure.

Browser vs. Server Deployments

The protocol is not browser-specific. The demo runs in a browser to prove the edge-canonical principle; production deployments can use Node.js, Deno, or any environment with Web Crypto API support. Browser deployments inherit browser risks (XSS, Spectre-class side channels). Server deployments avoid these but require their own hardening (network security, process isolation).
